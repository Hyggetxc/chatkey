# ChatKey PRD（MVP v1）

## 1. 文档信息

- 产品名称：ChatKey（暂定）
- 产品类型：macOS 菜单栏工具
- 当前阶段：功能规划 / MVP 定义
- 目标版本：MVP v1
- 更新时间：2026-03-28

## 2. GSD 总览

### Goal

做一个专注于聊天输入体验统一的 macOS 工具，让用户在不同 IM 或带聊天输入框的软件中，保持一致的输入习惯。

核心目标：

- 用户可以按应用分别定义聊天键位映射。
- 用户可以把 `Return`、`Command + Return` 等输入习惯统一下来，也可以自定义其他常见组合键。
- 工具尽量轻量、可理解，并支持简体中文 / 英文界面切换。

### Scope

MVP v1 只解决“聊天输入框按键映射”这一个问题，不做通用键盘改键器。

MVP v1 包含：

- 菜单栏常驻应用
- 按应用配置规则
- 预设 `Return` 与 `Command + Return` 的聊天映射模板
- 支持用户自定义触发键与目标动作
- 监听前台应用并按规则重映射
- 本地持久化规则
- 辅助功能权限引导
- 简体中文 / 英文界面切换
- GitHub Releases 更新检查
- 简单日志与状态提示

MVP v1 不包含：

- 全键盘任意按键 remap
- 浏览器域名级规则
- 云同步
- 脚本自动化平台
- 点击 UI 按钮发送消息
- OCR、图像识别、复杂输入框识别

### Success

MVP 成功标准：

- 用户能为至少 3 个常用聊天应用创建独立规则。
- 用户能稳定实现“我按下 A，软件收到 B”的映射。
- 用户能一眼知道当前规则是否生效、为何失败。
- 对不兼容应用可单独关闭，不影响其他软件。

## 3. 背景与问题

用户在不同 IM、协作软件、AI 聊天工具中的发送与换行快捷键不统一。

典型差异：

- 软件 A：`Return = 发送`，`Shift + Return = 换行`
- 软件 B：`Return = 换行`，`Command + Return = 发送`
- 软件 C：可配置，但入口深、记忆成本高

这会导致：

- 用户在不同软件间频繁误发送
- 需要反复切换肌肉记忆
- 对高频聊天用户造成明显认知负担

## 4. 目标用户

### 主要用户

- 高频使用微信、飞书、Slack、Telegram、Discord、企业 IM 的用户
- 同时使用多个聊天软件的知识工作者
- 对输入习惯一致性要求高的 macOS 用户

### 典型特征

- 每天跨多个聊天工具沟通
- 对快捷键敏感
- 讨厌误发送
- 不希望配置复杂自动化工具

## 5. 核心使用场景

### 场景 1：统一习惯

用户希望：

- `Return` 总是表示“换行”
- `Command + Return` 总是表示“发送”

即使底层不同应用真正需要的是：

- `Shift + Return`
- `Return`
- `Command + Return`

也由 ChatKey 帮用户转换。

### 场景 2：按应用差异化

用户在微信里需要：

- `Return -> Shift + Return`
- `Command + Return -> Return`

用户在另一个工具里需要：

- `Return -> Return`
- `Command + Return -> Command + Return`

ChatKey 需要支持每个应用单独配置。

### 场景 2.1：用户有自己的组合键习惯

除了默认的：

- `Return`
- `Command + Return`

用户还可能希望使用：

- `Shift + Return`
- `Option + Return`
- `Control + Return`

因此 ChatKey 不应只固定两种触发键，而应允许用户在“常见聊天组合键”范围内自定义触发方式。

### 场景 3：快速排错

如果某应用规则未生效，用户需要快速知道：

- 当前前台应用是什么
- 是否命中规则
- 规则是否启用
- 是否缺少辅助功能权限

## 6. 产品定位

ChatKey 是一个“聊天键位统一器”，不是：

- Karabiner 的替代品
- Keyboard Maestro 的替代品
- Hammerspoon 的替代品

定位特点：

- 专注一个问题
- 对中文用户友好
- 支持英文界面切换
- 低学习成本
- 为聊天场景而生

## 7. MVP 功能清单

### 7.1 菜单栏状态

支持：

- 显示当前运行状态
- 一键启用 / 暂停
- 显示当前前台应用
- 快速进入规则设置页

### 7.2 应用规则管理

支持：

- 展示应用名称、图标、Bundle ID
- 新增规则
- 编辑规则
- 启用 / 停用单条规则
- 删除规则
- 从当前前台应用快速创建规则

### 7.3 自定义键位映射

每条规则至少支持配置一组或多组“触发键 -> 目标动作”。

MVP 默认预设：

- 触发键 1：`Return`
- 触发键 2：`Command + Return`

用户可以在预设基础上改成别的支持组合。

### 7.4 触发键与目标动作集合

MVP v1 支持以下触发键候选：

- `Return`
- `Shift + Return`
- `Command + Return`
- `Option + Return`
- `Control + Return`

MVP v1 支持以下目标动作：

- 不处理
- `Return`
- `Shift + Return`
- `Command + Return`
- `Option + Return`
- `Control + Return`

备注：

- 第一版先聚焦 `Return` 相关组合，覆盖聊天发送 / 换行主场景。
- 后续版本再扩展为更多非 `Return` 组合。

### 7.5 权限与诊断

支持：

- 检查辅助功能权限
- 引导用户前往系统设置授权
- 显示最近一次规则命中结果
- 显示失败原因

### 7.6 语言切换

支持：

- 简体中文
- English
- 在设置页手动切换界面语言

MVP 建议：

- 默认跟随系统语言
- 用户可手动覆盖系统语言
- 切换后立即刷新主要界面文案

### 7.7 更新检查

支持：

- 自动检查新版本
- 手动检查新版本
- 发现新版本后展示提示
- 一键跳转 GitHub Releases 下载页

MVP 建议：

- 默认开启自动检查更新
- 用户可在设置中关闭“自动检查更新”
- 不做应用内自动下载与覆盖安装
- 不做静默更新

## 8. 规则模型设计

每个应用对应一条或多条规则，MVP v1 先采用“每应用一条规则”的简化模型。

### 8.1 Rule 结构

```json
{
  "id": "uuid",
  "appName": "WeChat",
  "bundleId": "com.tencent.xinWeChat",
  "isEnabled": true,
  "matchMode": "bundleId",
  "mappings": [
    {
      "trigger": "return",
      "output": "shiftReturn"
    },
    {
      "trigger": "commandReturn",
      "output": "return"
    }
  ],
  "notes": "Return 换行，Cmd+Return 发送",
  "updatedAt": "2026-03-28T12:00:00Z"
}
```

### 8.2 Trigger 枚举

- `return`
- `shiftReturn`
- `commandReturn`
- `optionReturn`
- `controlReturn`

### 8.3 Output 枚举

- `none`
- `return`
- `shiftReturn`
- `commandReturn`
- `optionReturn`
- `controlReturn`

## 9. 核心交互设计

### 9.1 首次启动

流程：

1. 打开应用
2. 检查辅助功能权限
3. 若未授权，展示中文引导页
4. 授权完成后进入规则页

### 9.2 新建规则

推荐流程：

1. 用户点击“从当前应用创建规则”
2. 系统读取当前前台应用名称、图标、Bundle ID
3. 用户设置：
   - `Return` 映射到什么
   - `Command + Return` 映射到什么
4. 保存后立即生效

### 9.3 日常使用

用户一般不需要打开主界面，只需要：

- 菜单栏确认是否启用
- 偶尔切换开关
- 某个应用不对时进入规则页微调

## 10. 非功能需求

### 10.1 性能

- 常驻占用低
- 键盘监听响应应接近实时
- 不明显增加输入延迟

### 10.2 易用性

- 全中文界面
- 不使用技术术语轰炸用户
- 关键状态可见

补充：

- 需支持中英文文案切换
- 关键术语在中英文下保持一致映射，减少理解偏差
- 更新提示应明确告诉用户“将跳转 GitHub 下载最新版覆盖安装”

### 10.3 安全性

- 不上传输入内容
- 不记录聊天内容
- 只记录规则命中和错误日志

## 11. 风险与边界

### 风险 1：不同应用行为不一致

部分应用的“发送”不是简单快捷键，而是内部逻辑或按钮行为。

应对：

- MVP 先只支持快捷键重映射
- 后续版本再加入“Accessibility 点击发送按钮”兜底方案

### 风险 2：系统安全限制

在 Secure Input 等场景下，事件拦截可能失败。

应对：

- 明确提示当前不可接管
- 不做静默失败

### 风险 3：网页聊天框复杂度高

浏览器里多个站点共用同一个宿主应用，按应用匹配不够细。

应对：

- MVP 不承诺网页域名级支持
- 后续再评估窗口标题 / 浏览器扩展方案

## 12. 验收标准

功能验收：

- 可新增、编辑、删除应用规则
- 可为触发键和目标动作分别设置组合
- 默认可快速使用 `Return` / `Command + Return` 模板
- 前台切换应用后，命中不同规则
- 菜单栏可启用 / 暂停总开关
- 未授权时可正确引导
- 可在设置中切换中文 / 英文界面
- 默认自动检查更新，且用户可关闭
- 支持手动检查更新并跳转 GitHub Releases

体验验收：

- 普通用户无需理解事件拦截原理，也能完成配置
- 某个应用不生效时，能快速定位原因

- 前台应用识别

## 14. 当前结论

这个产品的核心，不应该是“固定把 Return 改成换行”，而应该是：

“允许用户按应用定义自己的聊天输入习惯，并由系统自动转换成该应用能理解的快捷键行为。”

这会让产品更通用，也更符合真实聊天软件的差异现状。
